# ADR-007 — Embedding Provider and Model

## Status

Accepted

## Context

MemoryOps AI requires a unified vector representation space for storing memories and executing semantic similarity queries. 

The database schema and memory repository are configured with a fixed vector size of 1536 dimensions. However, dimensional equality alone does not guarantee semantic compatibility. To execute meaningful cosine similarity calculations, query embeddings and stored memory embeddings must occupy the same vector space, generated by the same underlying model and version.

Prior to this decision, the codebase lacked an authoritative choice of embedding provider, model identity, or runtime configuration strategy, preventing end-to-end read-path integration.

## Decision

To support the Phase 2 MVP, we lock the following embedding architecture:

*   **Embedding Provider:** OpenAI
*   **Embedding Model:** `text-embedding-3-small`
*   **Vector Dimensions:** 1536
*   **Abstraction Boundary:** The existing `EmbeddingService` Abstract Base Class (ABC) remains the system boundary.
*   **Credential Source:** The OpenAI API key must be supplied at runtime using the environment variable `OPENAI_API_KEY`. Secrets must not be hardcoded or committed to source control.
*   **Model Invariant:** Both stored memory embeddings and query embeddings must occupy the same `text-embedding-3-small` space. 

To maintain simplicity and avoid premature optimization in the Phase 2 MVP:
*   Do not introduce a multi-provider router.
*   Do not introduce a model fallback chain.
*   Do not implement provider-neutral dynamic configuration for switching models at runtime.

## Failure Semantics

The concrete OpenAI embedding adapter must propagate all provider, HTTP client, connection, authentication, and rate-limit exceptions out of `generate_embedding(...)`. 

*   The adapter must not silently swallow exceptions.
*   The adapter must not substitute failures with `[]`, `None`, zero vectors, or cached placeholder values.
*   No retry policy is required within the adapter for the Phase 2 MVP.
*   The `RetrievalCoordinator` retains sole ownership of the narrow exception boundary and fallback logic: if the adapter throws during embedding generation, the coordinator catches it, sets `query_embedding = None`, and degrades the turn to `RetrievalMode.FALLBACK`.

## Input Contract

The concrete adapter must validate the input text string before calling the OpenAI provider API:

*   Empty strings and whitespace-only strings must be rejected with a `ValueError`.
*   Valid text containing leading or trailing whitespace must be forwarded unchanged.
*   The adapter must not lowercase, tokenize, normalize (e.g. NFKC), or otherwise preprocess the text. Preprocessing belongs to the Retriever's lexical match pipeline.
*   The adapter must not silently truncate the text.

## Response Validation

The concrete adapter must enforce strict validation on the provider's returned payload:

*   A successful generation must yield a Python `List[float]` of exactly 1536 finite elements.
*   Any malformed payloads (missing fields, null embedding values, wrong dimensions, non-finite values like `NaN`, `+inf`, or `-inf`) must raise an exception.
*   The adapter must not pad, truncate, project, or otherwise coerce a malformed response into 1536 dimensions.

## Alternatives Considered

### Provider-Neutral Arbitrary Model Selection
Permits configuring arbitrary model strings in environment variables. Rejected for the Phase 2 MVP because it introduces model/space drift risks, configuration churn, and complex validation requirements.

### Mock/Deterministic Production Embeddings
Generating fake, hash-based, or zero vectors. Rejected because it does not represent real semantic similarity capability and violates read-path validity.

### Different Models for Memory and Query Embeddings
Using different models for stored memories and incoming queries. Rejected because cross-model cosine similarity comparisons are mathematically invalid.

### Selecting Another 1536-Dimensional Model
Selecting a different model (e.g., Voyage or Cohere 1536-dim models) solely because the dimension matches. Rejected because it violates vector-space compatibility with `text-embedding-3-small`.

## Consequences

*   **Positive:** Guaranteed vector-space compatibility, predictable cosine similarity calculations, clear provider boundaries, and a clean path to implement the adapter in Step 8F.
*   **Trade-off:** OpenAI becomes an external runtime dependency. Active read paths require a valid `OPENAI_API_KEY` and network connection, or they degrade to fallback mode.
*   **Future Migration:** Switching embedding models later will require a database migration to alter vector sizes and a batch task to re-embed all existing memory records.

## Non-Goals

This ADR does not define:
*   The final assistant response LLM provider.
*   Multi-provider embedding routing or auto-selection.
*   Automatic retries or backoff logic.
*   Background re-embedding workers or migration tooling.
*   Vector database index creation.
*   Gateway dependency injection wiring.
