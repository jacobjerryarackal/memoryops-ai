# ADR-008 — Provider-Agnostic Embedding Architecture

## Status

Accepted

## Context

Prior to this decision, [ADR-007](file:///d:/AI/memoryops-ai/infra/adr/ADR-007-embedding-provider-and-model.md) locked the system to a single embedding provider (OpenAI) and a single model (`text-embedding-3-small` with 1536 dimensions) to avoid premature optimization. 

To support diverse runtime configurations without code modifications, we require a provider-agnostic subsystem that supports multiple providers (OpenAI, Google Gemini, and Offline Fallback) behind a unified interface. However, the database schema (`VECTOR(1536)`) and domain validation models remain frozen, requiring any alternative provider models to conform to the 1536-dimensional vector space contract.

## Decision

We evolve the embedding subsystem to support runtime provider selection using the `EMBEDDING_PROVIDER` environment variable. The following architecture is adopted:

1. **Abstraction Boundary:** The abstract interface `EmbeddingService` remains the system boundary.
2. **Provider Factory:** A factory function `get_embedding_service` resolves and instantiates the active provider class based on the environment configuration:
   - `openai` (default): Instantiates `OpenAIEmbeddingService` (using `text-embedding-3-small`, 1536-dim).
   - `gemini`: Instantiates `GeminiEmbeddingService` (using `text-embedding-004`, 768-dim natively).
   - `fallback`: Instantiates `FallbackEmbeddingService` (triggers immediate offline fallback).
3. **Zero-Padding for Dimensionality Invariance:** Since the database vector index is configured for 1536 dimensions and Pydantic models validate `len(v) == 1536`, `GeminiEmbeddingService` will pad its native 768-dimensional output with 768 trailing zeros to yield a 1536-dimensional vector. 
4. **Mathematical Equivalence of Zero-Padding:** Appending trailing zeros preserves the dot product and vector magnitudes of vectors generated by the same model, which ensures that cosine similarity calculations between padded vectors are mathematically identical to those on the original 768-dimensional space.

### Critical Warning on Cross-Provider Embedding Interchangeability

> [!WARNING]
> **Switching embedding providers (`EMBEDDING_PROVIDER`) requires re-embedding all stored vectors (or rebuilding the vector store).**
>
> Embeddings from different providers (e.g., OpenAI vs. Gemini) occupy entirely different semantic vector spaces and are **not** semantically interchangeable, even if their dimensionalities are normalized to 1536 via padding. Storing vectors from provider A and querying them using provider B will result in invalid cosine similarity comparisons, producing corrupt or meaningless retrieval rankings.

## Failure Semantics

- Concrete provider implementations must propagate connection, timeout, authentication, and HTTP status code exceptions out of `generate_embedding(...)`. They must not silently swallow errors or return dummy vectors.
- The `FallbackEmbeddingService` explicitly raises a `RuntimeError` on invocation, causing the `RetrievalCoordinator` to catch the exception and fall back to lexical retrieval (`retrieval_mode = RetrievalMode.FALLBACK`, `query_embedding = None`).

## Input Contract

All concrete providers must enforce the same input validation rules:
- Empty strings and whitespace-only strings must be rejected with a `ValueError`.
- Valid text containing leading or trailing whitespace must be forwarded unchanged.
- Providers must not preprocess (lowercase, tokenize, normalize) the text.

## Response Validation

- The factory and concrete services must ensure that successful generations yield a Python `List[float]` of exactly 1536 finite elements.
- Malformed payloads (missing fields, wrong dimensions, non-finite values like `NaN` or `inf`) must raise an exception to trigger the retrieval fallback boundary.

## Alternatives Considered

### Database Schema Migration to 768 Dimensions
Altering the database schema to dynamically support 768 dimensions when Gemini is selected. Rejected because it violates frozen Phase 2 contracts and requires schema/index alterations at runtime, introducing substantial operational risk.

### Cross-Model Cosine Comparisons
Allowing querying of OpenAI-embedded memories using Gemini-embedded queries. Rejected because cross-model calculations are mathematically invalid.

---

## Consequences

- **Positive:** Full flexibility to swap providers at runtime without code changes; mathematical invariance of ranking is preserved; zero database migrations or schema alterations required.
- **Trade-off:** Operators must explicitly manage database clears and re-embeddings when switching `EMBEDDING_PROVIDER` in production.
